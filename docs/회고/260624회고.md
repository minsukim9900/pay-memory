# BaseTimeEntity에서 LocalDateTime 대신 Instant를 사용한 이유

프로젝트에서 모든 엔티티의 생성 시각과 수정 시각을 공통으로 관리하기 위해 `BaseTimeEntity`를 만들었습니다.

```java
package site.paymemory.global.entity;

import jakarta.persistence.Column;
import jakarta.persistence.EntityListeners;
import jakarta.persistence.MappedSuperclass;
import lombok.Getter;
import org.springframework.data.annotation.CreatedDate;
import org.springframework.data.annotation.LastModifiedDate;
import org.springframework.data.jpa.domain.support.AuditingEntityListener;

import java.time.Instant;

@Getter
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseTimeEntity {

    @CreatedDate
    @Column(nullable = false, updatable = false)
    private Instant createdAt;

    @LastModifiedDate
    @Column(nullable = false)
    private Instant updatedAt;
}
```

JPA Auditing을 적용한 예제를 찾아보면 생성 시각과 수정 시각의 타입으로 `LocalDateTime`을 사용하는 경우가 많습니다.

하지만 이번 프로젝트에서는 `createdAt`과 `updatedAt`의 타입으로 `LocalDateTime`이 아닌 `Instant`를 사용했습니다.

단순히 `Instant`가 더 좋은 타입이기 때문에 선택한 것은 아닙니다.

`createdAt`과 `updatedAt`이 나타내야 하는 값이 **특정 지역에서 보이는 날짜와 시간**이 아니라, **엔티티가 실제로 생성되고 수정된 순간**이라고 판단했기 때문입니다.

---

## 시간 타입을 왜 고민하게 되었을까?

웹 애플리케이션은 하나의 환경에서만 동작하지 않습니다.

일반적으로 다음과 같은 여러 구성 요소가 함께 동작합니다.

```text
데이터베이스
백엔드 서버
프론트엔드
사용자의 브라우저
```

각 구성 요소는 서로 다른 기본 시간대를 사용할 수 있습니다.

예를 들어 개발자의 로컬 환경은 `Asia/Seoul`이고, 운영 서버는 `UTC`일 수 있습니다. 데이터베이스 서버가 또 다른 시간대 설정을 사용할 수도 있습니다.

이때 다음과 같이 시간대가 없는 값만 전달한다면 문제가 발생할 수 있습니다.

```text
2026-06-24 10:00:00
```

이 값만으로는 다음 내용을 판단할 수 없습니다.

- 서울 기준 오전 10시인지
- UTC 기준 오전 10시인지
- 서버가 위치한 지역의 오전 10시인지

즉, 날짜와 시간만으로는 해당 값이 실제 타임라인상의 어느 순간을 나타내는지 알 수 없습니다.

이러한 모호함을 줄이기 위해서는 먼저 시간 값의 의미를 구분해야 합니다.

```text
지역의 날짜와 시간이 중요한 값인가?
사건이 실제로 발생한 순간이 중요한 값인가?
```

`BaseTimeEntity`의 `createdAt`과 `updatedAt`은 후자에 해당합니다.

## LocalDateTime은 어떤 타입일까?

![같은 LocalDateTime을 서로 다른 시간대로 해석한 결과](https://velog.velcdn.com/images/mindoli/post/327b5719-e18c-4bdd-ba59-3c1d4094e9a6/image.png)

`LocalDateTime`은 날짜와 시간을 함께 표현하는 타입입니다.

```text
2026-06-24T10:00:00
```

하지만 `LocalDateTime`에는 시간대와 UTC 오프셋이 포함되어 있지 않습니다.

Java 공식 문서에서도 `LocalDateTime`을 시간대가 없는 날짜와 시간으로 설명합니다. 또한 오프셋이나 시간대와 같은 추가 정보 없이는 타임라인상의 한 시점을 나타낼 수 없다고 설명합니다.

예를 들어 다음과 같은 `LocalDateTime`이 있다고 가정하겠습니다.

```java
LocalDateTime localDateTime =
        LocalDateTime.of(2026, 6, 24, 10, 0);
```

이 값에 UTC를 적용하면 다음과 같은 `Instant`가 만들어집니다.

```java
Instant utcInstant = localDateTime
        .atZone(ZoneId.of("UTC"))
        .toInstant();

System.out.println(utcInstant);
// 2026-06-24T10:00:00Z
```

같은 값에 서울 시간대를 적용하면 결과가 달라집니다.

```java
Instant seoulInstant = localDateTime
        .atZone(ZoneId.of("Asia/Seoul"))
        .toInstant();

System.out.println(seoulInstant);
// 2026-06-24T01:00:00Z
```

두 값 모두 지역 시간으로는 `2026년 6월 24일 오전 10시`입니다.

하지만 첫 번째 값은 UTC 기준 오전 10시이고, 두 번째 값은 서울 기준 오전 10시입니다. 실제 타임라인에서는 두 값 사이에 9시간의 차이가 존재합니다.

```text
2026-06-24 10:00을 UTC로 해석
→ 2026-06-24T10:00:00Z

2026-06-24 10:00을 Asia/Seoul로 해석
→ 2026-06-24T01:00:00Z
```


따라서 `LocalDateTime`을 시스템의 생성 시각이나 수정 시각으로 사용하려면 다음과 같은 별도의 규칙이 필요합니다.

> 데이터베이스에 저장된 모든 LocalDateTime은 Asia/Seoul 기준으로 해석합니다.

서비스 전체가 하나의 시간대만 사용하고 해당 규칙을 일관되게 관리한다면 `LocalDateTime`을 사용할 수도 있습니다.

다만 `LocalDateTime` 값 자체에는 이 규칙이 포함되어 있지 않습니다.

값을 저장하고 조회하는 모든 구성 요소가 동일한 규칙을 알고 있어야 하며, 서버나 데이터베이스의 시간대 설정이 달라지면 의도하지 않은 해석이 발생할 수 있습니다.

> 참고: [Oracle Java API - LocalDateTime](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/time/LocalDateTime.html)

## Instant는 어떤 타입일까?

![하나의 Instant를 서로 다른 지역 시간으로 표현한 결과](https://velog.velcdn.com/images/mindoli/post/b8ae8407-d0a6-423f-93ad-ff9c65e7974a/image.png)

Java 공식 문서에서는 `Instant`를 **타임라인상의 한 시점**이라고 설명합니다.

`Instant`는 Java Epoch인 `1970-01-01T00:00:00Z`를 기준으로 경과한 초와 나노초를 이용해 하나의 순간을 표현합니다.

```text
2026-06-24T01:00:00Z
```

여기에서 `Z`는 UTC 오프셋이 `+00:00`이라는 의미입니다.

위 시점을 서울 시간으로 표현하면 다음과 같습니다.

```text
2026-06-24T10:00:00+09:00
```

두 값은 표현되는 날짜와 시간이 다르지만 실제로는 같은 순간을 나타냅니다.

```java
Instant instant =
        Instant.parse("2026-06-24T01:00:00Z");

ZonedDateTime seoulDateTime =
        instant.atZone(ZoneId.of("Asia/Seoul"));

System.out.println(seoulDateTime);
// 2026-06-24T10:00+09:00[Asia/Seoul]
```

`Instant`는 `UTC`나 `Asia/Seoul`과 같은 시간대 ID를 저장하는 타입은 아닙니다.

`Instant` 자체는 시간대와 관계없이 타임라인상의 하나의 순간을 나타냅니다. 해당 순간을 특정 지역의 날짜와 시간으로 표현할 때 `ZoneId`를 적용합니다.

따라서 `Instant`를 단순히 “UTC 시간을 저장하는 타입”이라고 설명하기보다는 다음과 같이 설명하는 것이 더 정확합니다.

> Instant는 UTC Epoch를 기준으로 타임라인상의 하나의 순간을 나타내는 타입입니다.


> 참고: [Oracle Java API - Instant](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/time/Instant.html)

## 웹 애플리케이션에서는 시간을 어떻게 관리해야 할까?

시간 값을 무조건 하나의 타입으로 통일하는 것이 정답은 아닙니다.

먼저 해당 값이 무엇을 의미하는지 구분해야 합니다.

### 지역의 날짜와 시간이 중요한 경우

다음과 같은 값은 특정 지역에서 보이는 날짜나 시간 자체가 중요합니다.

```text
매일 오전 9시에 영업을 시작합니다.
서울 기준으로 6월 24일 오후 2시에 수업을 시작합니다.
사용자의 생일은 6월 24일입니다.
```

이러한 값에는 상황에 따라 다음 타입을 사용할 수 있습니다.

```text
LocalDate
LocalTime
LocalDateTime
LocalDateTime + ZoneId
```

예를 들어 생년월일은 특정 순간이 아니라 날짜 자체가 중요하므로 `LocalDate`가 자연스럽습니다.

특정 지역의 예약 일시라면 `LocalDateTime`만 저장하기보다 해당 예약이 어느 지역 기준인지 함께 관리해야 할 수 있습니다.

### 사건이 발생한 순간이 중요한 경우

다음과 같은 값은 지역에서 몇 시로 보이는지보다 실제로 사건이 발생한 순간이 중요합니다.

```text
결제가 승인된 순간
사용자가 로그인한 순간
주문이 생성된 순간
엔티티가 수정된 순간
```

이러한 값은 타임라인상의 하나의 시점으로 관리하는 것이 자연스럽습니다.

Java에서는 이를 `Instant`로 표현할 수 있습니다.

```java
Instant occurredAt = Instant.now();
```

이번 프로젝트의 `createdAt`과 `updatedAt`도 사건이 발생한 순간에 해당합니다.

따라서 다음과 같이 저장과 표시의 책임을 분리했습니다.

```text
시스템 내부의 사건 발생 시점: Instant
사용자에게 보여주는 지역 시간: Instant + ZoneId
```

이 원칙을 한 문장으로 정리하면 다음과 같습니다.

> 시스템 내부에서는 사건의 시점을 명확하게 관리하고, 사용자에게 보여줄 때 필요한 지역 시간으로 변환합니다.

## BaseTimeEntity에는 왜 Instant를 사용했을까?

`createdAt`과 `updatedAt`은 엔티티의 상태가 변경된 시점을 기록하는 Auditing 필드입니다.

```text
createdAt: 엔티티가 생성된 순간
updatedAt: 엔티티가 마지막으로 수정된 순간
```

두 필드에서 중요한 것은 특정 지역에서 몇 시로 보이는지가 아닙니다.

실제로 어느 순간에 데이터가 생성되고 수정되었는지가 중요합니다.

### 시스템 이벤트의 발생 시점을 표현할 수 있습니다

`createdAt`은 엔티티가 생성된 순간을 나타냅니다.

`updatedAt`은 엔티티가 마지막으로 수정된 순간을 나타냅니다.

두 값은 사용자가 직접 입력하는 지역 시간이 아니라 시스템에서 발생한 이벤트의 타임스탬프입니다.

따라서 지역의 날짜와 시간을 표현하는 `LocalDateTime`보다 타임라인상의 한 시점을 나타내는 `Instant`가 필드의 의미에 더 잘 맞는다고 판단했습니다.

### 사건의 발생 순서를 비교할 수 있습니다

`Instant`는 타임라인상의 위치를 나타내기 때문에 두 사건의 선후 관계를 비교할 수 있습니다.

```java
if (createdAt.isBefore(updatedAt)) {
    // createdAt이 updatedAt보다 이전 순간입니다.
}
```

서버나 사용자의 시간대가 달라져도 두 사건이 발생한 순서는 달라지지 않습니다.

### 서버의 기본 시간대에 대한 의존성을 줄일 수 있습니다

`LocalDateTime`을 하나의 시점으로 변환하려면 별도의 시간대 정보가 필요합니다.

이 과정에서 서버의 기본 시간대를 암묵적으로 사용하면 개발 환경과 운영 환경의 설정 차이로 인해 서로 다른 결과가 발생할 수 있습니다.

반면 `Instant`는 이미 타임라인상의 하나의 시점을 나타냅니다.

따라서 엔티티의 생성 시각과 수정 시각을 다룰 때 서버의 기본 시간대를 기준으로 의미를 해석할 필요가 없습니다.

### 저장 기준과 표시 기준을 분리할 수 있습니다

시스템 내부에서는 `Instant`로 하나의 시점을 관리하고, 사용자에게 보여줄 때만 필요한 시간대를 적용할 수 있습니다.

```text
Entity와 내부 로직: Instant
API: Z 또는 UTC 오프셋이 포함된 시각
화면: 사용자에게 필요한 시간대로 변환
```

이렇게 저장 기준과 표시 기준을 분리하면 시간대가 다른 환경에서도 사건의 시점을 일관되게 관리할 수 있습니다.

## Spring Data JPA는 Instant를 지원할까?

Spring Data JPA는 `@CreatedDate`와 `@LastModifiedDate`를 이용해 엔티티의 생성 시각과 수정 시각을 자동으로 기록할 수 있도록 지원합니다.

```java
@CreatedDate
private Instant createdAt;

@LastModifiedDate
private Instant updatedAt;
```

Spring Data JPA 공식 문서에서도 `@CreatedDate`가 적용된 필드의 타입으로 `Instant`를 사용하는 예제를 확인할 수 있습니다.

```java
class Customer {

    @CreatedBy
    private User user;

    @CreatedDate
    private Instant createdDate;
}
```

공식 예제에서 `Instant`를 사용한다고 해서 모든 시간 필드에 반드시 `Instant`를 사용해야 한다는 의미는 아닙니다.

중요한 것은 프레임워크가 어떤 타입을 지원하는지가 아니라, 해당 필드가 어떤 의미를 표현하는지입니다.

이번 프로젝트의 `createdAt`과 `updatedAt`은 특정 지역의 날짜와 시간이 아니라 이벤트가 발생한 순간을 나타내므로 `Instant`를 선택했습니다.

> 참고: [Spring Data JPA Reference - Auditing](https://docs.spring.io/spring-data/jpa/reference/auditing.html)

## JPA Auditing은 어떻게 활성화할까?

`@CreatedDate`와 `@LastModifiedDate`를 사용하려면 JPA Auditing을 활성화해야 합니다.

```java
@Configuration
@EnableJpaAuditing
public class JpaAuditingConfig {
}
```

그리고 Auditing을 적용할 엔티티에는 `AuditingEntityListener`를 등록해야 합니다.

```java
@EntityListeners(AuditingEntityListener.class)
```

현재 작성한 `BaseTimeEntity`에는 `AuditingEntityListener`가 등록되어 있습니다.

```java
@Getter
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseTimeEntity {

    @CreatedDate
    @Column(nullable = false, updatable = false)
    private Instant createdAt;

    @LastModifiedDate
    @Column(nullable = false)
    private Instant updatedAt;
}
```

`BaseTimeEntity`를 상속하는 엔티티는 생성 및 수정 시각 필드를 공통으로 사용할 수 있습니다.

## 사용자에게 보여줄 때는 어떻게 처리할까?

![Instant를 저장하고 사용자 시간대로 표현하는 과정](https://velog.velcdn.com/images/mindoli/post/611e29d0-f762-4257-98be-2a5c086406c9/image.png)

시스템 내부에서 `Instant`를 사용한다고 해서 사용자에게도 UTC 형태의 문자열을 그대로 보여주어야 하는 것은 아닙니다.

예를 들어 다음 `Instant`가 있다고 가정하겠습니다.

```text
2026-06-24T01:00:00Z
```

한국 사용자에게 보여줄 때는 `Asia/Seoul` 시간대를 적용할 수 있습니다.

```java
ZonedDateTime createdAtInSeoul =
        createdAt.atZone(ZoneId.of("Asia/Seoul"));
```

결과는 다음과 같습니다.

```text
2026-06-24T10:00+09:00[Asia/Seoul]
```

`LocalDateTime` 형태가 필요하다면 다음과 같이 변환할 수 있습니다.

```java
LocalDateTime createdAtInSeoul =
        LocalDateTime.ofInstant(
                createdAt,
                ZoneId.of("Asia/Seoul")
        );
```

문자열로 포맷팅해야 한다면 `DateTimeFormatter`에 시간대를 명시할 수 있습니다.

```java
private static final DateTimeFormatter KOREA_DATE_TIME_FORMATTER =
        DateTimeFormatter
                .ofPattern("yyyy-MM-dd HH:mm:ss")
                .withZone(ZoneId.of("Asia/Seoul"));

String formattedCreatedAt =
        KOREA_DATE_TIME_FORMATTER.format(createdAt);
```

변환 결과는 다음과 같습니다.

```text
2026-06-24 10:00:00
```

이렇게 하면 시스템 내부의 `Instant`는 변경하지 않으면서 사용자에게는 한국 시간으로 표시할 수 있습니다.


핵심은 저장 기준과 표시 기준을 분리하는 것입니다.

```text
저장할 때: 사건이 발생한 하나의 시점으로 저장합니다.
표시할 때: 사용자 또는 서비스 정책에 맞는 시간대를 적용합니다.
```

---

## +09:00과 Asia/Seoul은 무엇이 다를까?

`+09:00`은 UTC보다 9시간 빠르다는 고정된 오프셋을 의미합니다.

```java
ZoneOffset offset = ZoneOffset.ofHours(9);
```

반면 `Asia/Seoul`은 서울 지역의 시간대 규칙을 나타내는 `ZoneId`입니다.

```java
ZoneId zoneId = ZoneId.of("Asia/Seoul");
```

두 값의 차이는 다음과 같습니다.

```text
+09:00
→ UTC와의 고정된 시간 차이입니다.

Asia/Seoul
→ 서울이라는 지역의 시간대 규칙을 나타냅니다.
```

현재 서울 시간은 UTC보다 9시간 빠르기 때문에 많은 상황에서 두 값을 적용한 결과가 같을 수 있습니다.

하지만 특정 지역을 의미하려는 경우에는 고정 오프셋보다 지역 기반의 `ZoneId`를 사용하는 편이 의도를 더 명확하게 나타냅니다.

서머타임을 사용하는 지역에서는 이 차이가 더 중요해집니다.

예를 들어 특정 지역의 시간대 규칙은 날짜에 따라 UTC 오프셋이 달라질 수 있지만, `+09:00`과 같은 고정 오프셋은 항상 같은 차이만 적용합니다.

> 참고: [Oracle Java API - ZoneId](https://docs.oracle.com/en/java/javase/21/docs/api/java.base/java/time/ZoneId.html)

---

## API에서는 어떤 형태로 전달하면 좋을까?

사건이 발생한 시점을 API로 전달할 때는 시간대 해석이 모호하지 않은 형태를 사용하는 것이 좋습니다.

예를 들어 `Instant`는 다음과 같이 `Z`가 포함된 ISO 8601 형식으로 표현할 수 있습니다.

```json
{
  "createdAt": "2026-06-24T01:00:00Z",
  "updatedAt": "2026-06-24T03:30:00Z"
}
```

이 값에는 UTC 오프셋이 명시되어 있으므로 클라이언트가 하나의 시점으로 해석할 수 있습니다.

반면 다음 값은 시간대나 오프셋이 포함되어 있지 않습니다.

```json
{
  "createdAt": "2026-06-24T10:00:00"
}
```

이 값만으로는 서울 기준 오전 10시인지, UTC 기준 오전 10시인지 알 수 없습니다.

따라서 사건의 발생 시점을 전달할 때는 다음과 같이 시간대 해석이 명확한 형태를 사용하는 편이 안전합니다.

```text
2026-06-24T01:00:00Z
2026-06-24T10:00:00+09:00
```

다만 API가 지역 일정이나 영업 시간처럼 지역 날짜와 시간 자체를 전달하는 경우에는 별도의 시간대 정보나 서비스 규칙이 필요할 수 있습니다.

## Instant를 사용하면 데이터베이스에도 UTC가 저장될까?

Java에서 `Instant`를 사용했다는 사실과 데이터베이스의 물리적인 저장 방식은 구분해서 생각해야 합니다.

```java
private Instant createdAt;
```

위 코드는 Java 도메인 모델에서 `createdAt`을 타임라인상의 하나의 시점으로 다루겠다는 의미입니다.

하지만 `Instant`를 사용했다고 해서 데이터베이스에 `UTC` 또는 `Asia/Seoul`과 같은 시간대 ID가 그대로 저장되는 것은 아닙니다.

Hibernate 6 공식 문서에서는 `Instant`를 `TIMESTAMP_UTC` SQL 타입에 매핑한다고 설명합니다.

반면 `LocalDateTime`은 `TIMESTAMP` 타입에 매핑합니다.

```text
Instant
→ TIMESTAMP_UTC

LocalDateTime
→ TIMESTAMP
```

다만 실제로 생성되는 데이터베이스 컬럼 타입과 시간대 처리 방식은 다음 요소에 따라 달라질 수 있습니다.

- 사용 중인 Hibernate 버전
- 데이터베이스의 종류와 버전
- Hibernate Dialect
- JDBC 드라이버
- 데이터베이스 세션 시간대
- Hibernate 시간대 관련 설정
- 데이터베이스가 지원하는 시간 정밀도

따라서 다음과 같이 단정하는 것은 피하는 것이 좋습니다.

> Instant를 사용하면 데이터베이스에 무조건 UTC 문자열이 저장됩니다.

다음과 같이 설명하는 편이 더 정확합니다.

> Java 도메인 모델에서 생성 시각과 수정 시각을 하나의 시점으로 다루기 위해 Instant를 선택했습니다. 실제 데이터베이스 저장 방식은 Hibernate와 데이터베이스의 매핑을 함께 확인해야 합니다.

프로젝트에서는 다음 항목을 직접 확인하는 것이 좋습니다.

```text
실제로 생성된 컬럼 타입은 무엇입니까?
저장 전과 조회 후의 Instant가 같은 시점을 나타냅니까?
초 이하 정밀도가 유지됩니까?
서버나 데이터베이스 세션 시간대가 달라도 같은 시점이 복원됩니까?
```

> 참고: [Hibernate ORM User Guide - Instant](https://docs.hibernate.org/orm/6.5/userguide/html_single/#basic-instant)  
> 참고: [Hibernate ORM User Guide - LocalDateTime](https://docs.hibernate.org/orm/6.5/userguide/html_single/#basic-localdatetime)

## 그렇다면 LocalDateTime은 언제 사용하면 좋을까?

`Instant`가 모든 날짜와 시간 문제의 정답인 것은 아닙니다.

`LocalDateTime`과 `Instant`는 어느 한쪽이 항상 더 좋은 타입이 아니라, 표현하려는 값의 의미에 따라 선택해야 하는 타입입니다.

| 표현하려는 값 | 고려할 타입 |
| --- | --- |
| 엔티티가 생성되거나 수정된 순간 | `Instant` |
| 결제가 승인된 순간 | `Instant` |
| 로그인이나 이벤트가 발생한 순간 | `Instant` |
| 생년월일처럼 날짜만 중요한 값 | `LocalDate` |
| 매일 오전 9시처럼 시각만 중요한 값 | `LocalTime` |
| 지역 날짜와 시간이 중요한 값 | `LocalDateTime`과 별도의 시간대 정책 |
| UTC 오프셋을 함께 표현해야 하는 값 | `OffsetDateTime` |
| 지역의 시간대 규칙까지 함께 다뤄야 하는 값 | `ZonedDateTime` |

예를 들어 다음 값들은 지역에서 보이는 날짜나 시간 자체가 중요합니다.

```text
매일 오전 9시에 영업을 시작합니다.
서울 기준으로 6월 24일 오후 2시에 수업을 시작합니다.
사용자의 생일은 6월 24일입니다.
```

이러한 값에는 `LocalDate`, `LocalTime`, `LocalDateTime` 등이 더 자연스러울 수 있습니다.

반면 다음 값들은 사건이 발생한 순간이 중요합니다.

```text
결제가 승인된 순간
로그인이 발생한 순간
주문이 생성된 순간
엔티티가 수정된 순간
```

이러한 값에는 `Instant`가 더 자연스럽습니다.

이번 `BaseTimeEntity`의 `createdAt`과 `updatedAt`도 후자에 해당합니다.

## 정리하면

이번에 `BaseTimeEntity`를 만들면서 생성 시각과 수정 시각을 어떤 타입으로 관리할지 고민했습니다.

처음에는 자연스럽게 `LocalDateTime`을 떠올렸습니다.

JPA Auditing 예제에서도 `LocalDateTime`을 사용하는 경우가 많고, 한국 서비스라면 `2026-06-24 10:00:00`처럼 사람이 읽기 쉬운 형태로 다루는 것이 더 직관적으로 느껴졌기 때문입니다.

하지만 `createdAt`과 `updatedAt`이 어떤 의미를 가지는지 다시 생각해 보니, 이 필드들은 사용자가 입력하는 지역 시간이 아니었습니다.

두 필드는 엔티티가 생성되고 수정된 **사건의 발생 시점**을 기록하는 값이었습니다.

`LocalDateTime`은 날짜와 시간을 표현하지만 시간대와 UTC 오프셋을 포함하지 않습니다.

따라서 별도의 시간대 정보 없이는 해당 값이 타임라인상의 어느 순간을 나타내는지 확정할 수 없습니다.

반면 `Instant`는 타임라인상의 하나의 시점을 나타냅니다.

두 타입을 비교하면 다음과 같은 트레이드 오프가 있습니다.

| 타입 | 장점 | 주의할 점 |
| --- | --- | --- |
| `LocalDateTime` | 사람이 읽기 쉽고 지역 날짜·시간을 표현하기 직관적입니다. | 시간대 정보가 없기 때문에 어느 지역 기준인지 별도의 규칙이 필요합니다. |
| `Instant` | 사건이 발생한 하나의 시점을 명확하게 표현할 수 있습니다. | 사용자에게 보여줄 때는 원하는 시간대로 변환해야 합니다. |

`LocalDateTime`을 사용하면 값 자체가 읽기 쉽고, 한국 시간 기준으로만 운영되는 서비스에서는 단순하게 사용할 수 있다는 장점이 있습니다.

반면 시간대 정보가 없기 때문에 이 값이 어느 지역 기준인지 별도의 규칙이 필요합니다.

개발 환경, 운영 서버, 데이터베이스, 배치 서버의 시간대 설정이 달라졌을 때 같은 값을 서로 다르게 해석할 가능성도 있습니다.

`Instant`를 사용하면 처음에는 `2026-06-24T01:00:00Z`처럼 사용자에게 익숙하지 않은 형태로 보일 수 있습니다.

화면에 표시할 때는 `Asia/Seoul` 같은 `ZoneId`를 적용해 변환해야 하므로 코드가 조금 더 필요합니다.

하지만 시스템 내부에서는 타임라인상의 하나의 시점으로 관리할 수 있고, 서버 시간대나 실행 환경에 덜 의존할 수 있다는 장점이 있습니다.

따라서 이번 프로젝트에서는 다음과 같은 기준을 적용했습니다.

```text
시스템 내부의 생성 및 수정 시점
→ Instant로 관리합니다.

사용자에게 보여주는 지역 시간
→ Instant에 필요한 ZoneId를 적용합니다.
```

결국 `Instant`가 모든 시간 문제의 정답이기 때문에 선택한 것은 아닙니다.

`createdAt`과 `updatedAt`이 답해야 하는 질문이 다음과 같았기 때문에 선택했습니다.

> 지역 시간으로 몇 시입니까?

가 아니라,

> 정확히 언제 발생했습니까?

라는 질문입니다.

이번 `BaseTimeEntity`의 `createdAt`과 `updatedAt`은 사용자가 보는 시간을 저장하기 위한 필드가 아니라, 데이터가 생성되고 수정된 순간을 기록하기 위한 Auditing 필드였습니다.

그래서 화면 표시의 편의성보다는 시스템 내부에서의 일관성과 명확성을 더 중요하게 보았습니다.

결과적으로 `LocalDateTime`보다 `Instant`가 Auditing 필드의 의미에 더 잘 맞는다고 판단했고, `BaseTimeEntity`에는 `Instant`를 사용했습니다.
