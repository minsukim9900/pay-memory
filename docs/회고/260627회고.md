# DTO에서 검증했는데 Entity에서도 검증해야 할까?

프로젝트에서 핵심 도메인 Entity를 구성하던 중, `nullable = false` 필드에 대한 검증을 어디까지 해야 하는지 고민하게 되었습니다.

이번 작업은 **페이메모리**라는 Spring Boot 기반 백엔드 프로젝트에서 ERD를 바탕으로 핵심 도메인 Entity와 Document를 구성하는 작업이었습니다.

구성한 주요 클래스는 다음과 같았습니다.

- `User`
- `TransactionCategory`
- `PaymentTransaction`
- `AiChatRoom`
- `AiChatMessage`
- `TransactionType`
- `MessageRole`
- `MessageType`

그중 PR 리뷰에서 주로 검증 관련 피드백을 받은 클래스는 다음과 같았습니다.

- `PaymentTransaction`
- `User`
- `AiChatRoom`
- `TransactionCategory`

처음에는 Request DTO에서 Bean Validation으로 검증하면 충분하다고 생각했습니다.

하지만 PR 리뷰를 통해 Entity 내부에서도 최소한의 방어 검증이 필요할 수 있다는 점을 다시 생각하게 되었습니다.

## 처음에는 DTO Validation만으로 충분하다고 생각했습니다

처음 생각은 단순했습니다.

> API 요청으로 들어오는 값은 Request DTO에서 검증하면 되지 않을까?

예를 들어 거래 금액을 생성하는 요청이 있다고 하면, Request DTO에서 다음처럼 검증할 수 있습니다.

```java
public record CreatePaymentTransactionRequest(
        @NotNull(message = "금액은 필수입니다.")
        @Positive(message = "금액은 0보다 커야 합니다.")
        Long amount
) {
}
```

이렇게 작성하면 사용자가 금액을 보내지 않았을 때는 `@NotNull`에서 막을 수 있고, 0 이하의 값을 보냈을 때는 `@Positive`에서 막을 수 있습니다.

즉, 잘못된 요청은 Entity까지 오기 전에 DTO Validation 단계에서 먼저 걸러질 수 있습니다.

그래서 처음에는 Entity 내부에 다시 다음과 같은 검증을 넣는 것이 중복처럼 느껴졌습니다.

```java
if (amount <= 0) {
    throw new IllegalArgumentException("금액은 0보다 커야 합니다.");
}
```

이미 DTO에서 검증한 값을 Entity에서 한 번 더 검증하는 것이 과한 방어 코드처럼 보였습니다.

특히 프로젝트 초반에는 대부분의 Entity 생성과 수정이 API 요청 흐름에서 시작된다고 생각했습니다.

흐름은 대략 다음과 같았습니다.

```text
Client Request
→ Controller
→ Request DTO Validation
→ Service
→ Entity
→ Repository
```

이 흐름만 놓고 보면 DTO Validation만으로도 충분해 보였습니다.

사용자가 잘못된 값을 보내면 DTO에서 막고, Entity에는 이미 검증된 값만 들어온다고 생각했습니다.

하지만 PR 리뷰를 받으면서 이 생각이 조금 바뀌었습니다.

Entity는 Controller 요청을 통해서만 생성되거나 수정되는 객체가 아니었습니다.

## 리뷰에서 지적받은 nullable = false 필드 검증

PR 리뷰에서 받은 핵심 피드백은 다음과 같았습니다.

> `nullable = false`로 선언된 필드라면 Entity 내부에서도 null 또는 잘못된 값으로 변경되지 않도록 방어할 필요가 있습니다.

특히 `PaymentTransaction`의 수정 메서드에서 이 문제가 드러났습니다.

거래 내역을 수정하는 `updateTransaction()` 메서드가 있을 때, 처음에는 DTO에서 이미 검증된 값이 들어온다고 생각해서 Entity 내부 검증을 크게 두지 않았습니다.

하지만 `updateTransaction()` 메서드는 꼭 Controller 요청을 통해서만 호출된다는 보장이 없었습니다.

Service 내부 로직에서 호출할 수도 있고, 테스트 코드에서 직접 호출할 수도 있었습니다.

나중에 배치 작업, 관리자 기능, 외부 API 연동, AI 분석 결과 기반 자동 생성 로직이 생기면 그쪽에서도 Entity를 직접 만들거나 수정할 수 있었습니다.

이때 DTO Validation을 거치지 않는 경로가 생길 수 있었습니다.

결국 리뷰에서 받은 지적은 단순히 “검증 코드를 더 넣자”는 의미가 아니었습니다.

조금 더 정확히 말하면 이런 의미였습니다.

> DTO는 요청 값을 검증하는 계층이고, Entity는 자기 자신의 유효한 상태를 지켜야 합니다.

이 관점으로 다시 보니 Entity 내부 검증이 단순 중복으로만 보이지 않았습니다.

DTO Validation과 Entity 내부 검증은 모두 “검증”이라는 이름을 가지고 있지만, 실제로는 책임이 달랐습니다.

## DTO Validation과 Entity 검증은 무엇이 다를까?

이번 리뷰를 통해 가장 크게 정리한 부분은 DTO Validation과 Entity 내부 검증의 역할 차이였습니다.

처음에는 둘 다 값을 검증한다는 점에서 비슷하게 느껴졌습니다.

하지만 목적이 달랐습니다.

DTO Validation은 사용자 입력을 검증하는 역할에 가까웠습니다.

예를 들어 사용자가 거래 금액을 보내지 않았거나, 빈 문자열을 보냈거나, 0 이하의 금액을 보냈을 때 이를 친절한 메시지로 알려주는 것이 목적이었습니다.

```java
public record CreatePaymentTransactionRequest(
        @NotNull(message = "금액은 필수입니다.")
        @Positive(message = "금액은 0보다 커야 합니다.")
        Long amount
) {
}
```

여기서 작성한 메시지는 클라이언트에게 전달될 수 있습니다.

사용자 입장에서 이해하기 쉬워야 하고, 어떤 필드를 어떻게 수정해야 하는지 알 수 있어야 했습니다.

반면 Entity 내부 검증은 사용자 입력 검증이라기보다 도메인 객체의 상태를 보호하는 역할에 가까웠습니다.

`PaymentTransaction`이라는 거래 내역이 존재한다면, 최소한 다음 값들은 정상적인 상태여야 했습니다.

- 거래 카테고리
- 거래 일시
- 사용처
- 거래 금액
- 거래 유형

이 값들이 비어 있거나 잘못된 상태라면, 그 객체는 도메인 관점에서 정상적인 거래 내역이라고 보기 어려웠습니다.

즉, DTO Validation은 “요청이 올바른가?”를 검증했습니다.

Entity 내부 검증은 “객체가 유효한 상태인가?”를 검증했습니다.

이번 프로젝트에서는 이 둘을 다음처럼 나누어 보기로 했습니다.

```text
DTO Validation
→ 사용자 입력 검증
→ 클라이언트에게 전달 가능한 메시지
→ API 요청 경계에서 잘못된 값을 차단

Entity 내부 검증
→ 도메인 상태 보호
→ 잘못된 객체 생성을 방지
→ DTO를 거치지 않는 경로에 대한 최소한의 방어

Service 비즈니스 예외
→ 존재하지 않는 사용자
→ 권한 없음
→ 중복 요청
→ 비즈니스 규칙 위반
```

이렇게 나누니 “DTO에서 검증했는데 Entity에서도 검증해야 하나?”라는 질문의 답이 조금 더 명확해졌습니다.

완전히 같은 검증을 두 번 한다기보다, 서로 다른 목적의 검증을 각 계층에 두는 것에 가까웠습니다.

## Entity는 Controller 요청으로만 변경되지 않습니다

![Entity는 다양한 경로에서 변경될 수 있다](https://velog.velcdn.com/images/mindoli/post/395c550a-057a-4433-9d9c-19bd01879959/image.png)

Entity 내부 검증이 필요하다고 느낀 가장 큰 이유는 Entity가 Controller 요청으로만 생성되거나 수정되지 않는다는 점이었습니다.

프로젝트가 작을 때는 대부분의 흐름이 단순했습니다.

```text
Client Request
→ Controller
→ Request DTO Validation
→ Service
→ Entity
→ Repository
```

이 흐름만 보면 DTO Validation이 모든 잘못된 값을 막아줄 것처럼 보였습니다.

하지만 프로젝트가 커지면 Entity를 사용하는 경로가 늘어납니다.

예를 들어 다음과 같은 경로가 생길 수 있습니다.

```text
Service 내부 로직
테스트 코드
배치 작업
관리자 기능
초기 데이터 생성 코드
외부 API 연동 결과 저장
AI 분석 결과 기반 자동 생성
```

페이메모리 프로젝트에서도 AI 채팅, 거래 내역 분석, 자동 분류 같은 흐름이 확장될 가능성이 있었습니다.

예를 들어 사용자가 직접 입력한 거래 내역은 Request DTO를 거칠 수 있습니다.

하지만 나중에 AI 분석 결과를 바탕으로 거래 내역을 자동 생성한다면 어떨까요?

그 로직은 사용자의 Request DTO를 그대로 거치지 않을 수 있습니다.

외부 결제 데이터나 카드 사용 내역을 연동한다면, 그 데이터도 항상 기존 Request DTO를 통해 들어온다고 보장하기 어려웠습니다.

테스트 코드도 마찬가지였습니다.

테스트에서 Entity를 직접 생성하거나 수정할 때 잘못된 값을 넣어도, Entity 내부에 방어 코드가 없다면 잘못된 상태의 객체가 만들어질 수 있었습니다.

이런 상황을 생각해보니 Entity 내부 검증은 단순히 “사용자 요청을 한 번 더 검증하는 코드”가 아니었습니다.

Entity가 자기 자신의 최소한의 무결성을 지키기 위한 코드였습니다.

## nullable = false와 도메인 무결성의 관계

![데이터 무결성을 지키는 3단계 방어선](https://velog.velcdn.com/images/mindoli/post/f98f2f3c-1491-48fd-90a0-082f21ee308f/image.png)

JPA Entity를 작성할 때 자주 사용하는 옵션 중 하나가 `nullable = false`입니다.

```java
@Column(name = "amount", nullable = false)
private Long amount;
```

처음에는 `nullable = false`를 설정하면 해당 필드는 null이 될 수 없다고 생각하기 쉽습니다.

하지만 정확히 보면 `nullable = false`는 주로 DB 스키마 레벨의 제약에 가깝습니다.

즉, 테이블 컬럼에 `NOT NULL` 제약을 걸어주는 역할을 합니다.

이 제약은 데이터베이스에 저장되는 순간에는 도움이 됩니다.

하지만 Java 객체가 메모리에서 생성되고 수정되는 동안에는 이야기가 조금 달랐습니다.

```java
paymentTransaction.updateTransaction(
        null,
        transactionAt,
        merchantName,
        amount,
        transactionType,
        includedInSpending,
        memo
);
```

Entity 내부에서 아무 검증도 하지 않는다면, 저장 직전까지는 잘못된 상태의 객체가 존재할 수 있습니다.

물론 DB에 저장할 때 예외가 발생할 수 있습니다.

하지만 그때 발생하는 예외는 도메인 규칙을 명확히 표현하는 예외라기보다, DB 제약 위반에 가까웠습니다.

또한 실패 시점도 늦어졌습니다.

객체를 잘못 만든 순간에 바로 알 수 있는 것이 아니라, flush나 commit 시점까지 가서야 문제가 드러날 수 있었습니다.

이번 리뷰를 통해 `nullable = false`를 다음처럼 바라보게 되었습니다.

```text
nullable = false
→ DB에 잘못된 값이 저장되지 않도록 막는 마지막 방어선

Entity 내부 검증
→ 도메인 객체가 잘못된 상태가 되지 않도록 막는 방어선
```

둘 중 하나만 있으면 충분하다고 보기보다, 서로 다른 위치에서 무결성을 지켜주는 장치라고 보는 것이 더 자연스러웠습니다.

`nullable = false`는 DB에 저장되는 데이터의 무결성을 지키는 장치였습니다.

Entity 내부 검증은 도메인 객체가 메모리상에서도 유효한 상태를 유지하도록 도와주는 장치였습니다.

## Long 대신 long을 선택한 이유

![DTO vs Entity 타입 선택 기준](https://velog.velcdn.com/images/mindoli/post/6ac26714-64d2-46f7-83a8-50d29380acb6/image.png)


검증과 함께 같이 고민한 부분이 타입 선택이었습니다.

처음 `PaymentTransaction`의 `amount` 필드는 다음과 같이 작성되어 있었습니다.

```java
@Column(name = "amount", nullable = false)
private Long amount;
```

하지만 리뷰에서 다음과 같은 피드백을 받았습니다.

- `amount`는 `nullable = false`이므로 항상 값이 있어야 합니다.
- `Long`을 사용하면 null 상태가 가능해집니다.
- 금액 계산 등 산술 연산이 자주 일어나는 필드이므로 primitive 타입인 `long`이 더 적절합니다.
- 생성자, 정적 팩토리 메서드, 수정 메서드 파라미터도 함께 `long`으로 변경하는 것이 좋습니다.

처음에는 `Long`을 사용하면 null이 들어왔을 때 직접 잡아서 에러 메시지를 줄 수 있다고 생각했습니다.

```java
if (amount == null) {
    throw new IllegalArgumentException("금액은 필수입니다.");
}
```

하지만 다시 생각해보니 null 여부를 친절하게 알려주는 책임은 Request DTO 쪽에 두는 것이 더 자연스러웠습니다.

외부 요청에서는 값이 누락될 수 있습니다.

그래서 Request DTO에서는 `Long`을 사용하고 `@NotNull`을 붙이는 것이 적절했습니다.

```java
public record CreatePaymentTransactionRequest(
        @NotNull(message = "금액은 필수입니다.")
        @Positive(message = "금액은 0보다 커야 합니다.")
        Long amount
) {
}
```

반면 Entity 내부에서는 금액이 없는 상태를 허용하지 않았습니다.

`PaymentTransaction`이 생성되었다면 `amount`는 반드시 존재해야 했습니다.

그래서 Entity에서는 `long`을 사용하는 쪽이 더 적절하다고 판단했습니다.

수정 후 Entity 필드는 다음과 같았습니다.

```java
@Column(name = "amount", nullable = false)
private long amount;
```

최종적으로 적용한 기준은 다음과 같았습니다.

```text
Request DTO
→ Long amount
→ @NotNull, @Positive로 요청 값 검증

Entity
→ long amount
→ null 상태 자체를 표현하지 않음
→ 0 이하 값은 Entity 내부에서 방어 검증
```

물론 primitive 타입에도 주의할 점은 있었습니다.

`long`의 기본값은 `0`입니다.

따라서 null은 표현할 수 없지만, 0이라는 잘못된 값이 들어갈 수는 있습니다.

거래 금액에서 0은 유효하지 않은 값이므로 Entity 내부에서 `amount <= 0` 검증은 여전히 필요하다고 봤습니다.

```java
if (amount <= 0) {
    throw new IllegalArgumentException("금액은 0보다 커야 합니다.");
}
```

즉, `long`을 사용한다고 검증이 사라지는 것은 아니었습니다.

다만 “값이 없음”이라는 상태를 Entity에서 표현하지 않도록 만들고, 도메인 규칙에 맞지 않는 값은 별도로 검증하는 방향으로 정리했습니다.

## Boolean 대신 boolean도 같은 기준으로 봐야 했습니다

같은 기준으로 `includedInSpending` 필드도 다시 보게 되었습니다.

이 필드는 지출 합계에 포함할지 여부를 나타내는 값입니다.

처음에는 다음처럼 Wrapper 타입인 `Boolean`을 사용할 수 있다고 생각했습니다.

```java
@Column(name = "included_in_spending", nullable = false)
private Boolean includedInSpending;
```

하지만 이 필드도 `nullable = false`였습니다.

도메인 관점에서도 지출 합계 포함 여부는 반드시 true 또는 false 중 하나로 결정되어야 했습니다.

값이 없는 상태, 즉 null은 의미가 애매했습니다.

```text
true
→ 지출 합계에 포함합니다.

false
→ 지출 합계에 포함하지 않습니다.

null
→ 포함한다는 건지, 포함하지 않는다는 건지 알 수 없습니다.
```

따라서 Entity에서는 `Boolean`보다 `boolean`이 더 적절하다고 판단했습니다.

```java
@Column(name = "included_in_spending", nullable = false)
private boolean includedInSpending;
```

이 기준은 `amount`와 비슷했습니다.

외부 요청에서 값이 누락되었는지 확인해야 한다면 DTO에서는 `Boolean`을 사용할 수 있습니다.

예를 들어 사용자가 반드시 포함 여부를 보내야 한다면 Request DTO에서는 다음처럼 작성할 수 있습니다.

```java
public record UpdateIncludedInSpendingRequest(
        @NotNull(message = "지출 합계 포함 여부는 필수입니다.")
        Boolean includedInSpending
) {
}
```

반면 Entity에서는 null 상태가 존재하면 안 되므로 `boolean`을 사용했습니다.

```text
Request DTO
→ Boolean
→ 누락 여부를 검증할 수 있어야 함

Entity
→ boolean
→ null 상태를 허용하지 않음
```

이번 리뷰를 통해 Wrapper 타입과 primitive 타입을 선택할 때 단순히 습관적으로 고르면 안 된다는 점을 다시 느꼈습니다.

특히 `nullable = false`이고 도메인 내부에서 null 상태가 의미 없다면, primitive 타입이 더 명확한 선택일 수 있었습니다.

## PaymentTransaction에 적용한 검증

최종적으로 `PaymentTransaction`에는 필수 필드 검증을 위한 private 메서드를 추가했습니다.

수정 메서드는 다음과 같은 형태가 되었습니다.

```java
@Column(name = "amount", nullable = false)
private long amount;

@Column(name = "included_in_spending", nullable = false)
private boolean includedInSpending;

public void updateTransaction(
        TransactionCategory transactionCategory,
        Instant transactionAt,
        String merchantName,
        long amount,
        TransactionType transactionType,
        boolean includedInSpending,
        String memo
) {
    validateTransactionFields(
            transactionCategory,
            transactionAt,
            merchantName,
            amount,
            transactionType
    );

    this.transactionCategory = transactionCategory;
    this.transactionAt = transactionAt;
    this.merchantName = merchantName;
    this.amount = amount;
    this.transactionType = transactionType;
    this.includedInSpending = includedInSpending;
    this.memo = memo;
}
```

검증 메서드는 다음과 같이 작성했습니다.

```java
private void validateTransactionFields(
        TransactionCategory transactionCategory,
        Instant transactionAt,
        String merchantName,
        long amount,
        TransactionType transactionType
) {
    Objects.requireNonNull(transactionCategory, "거래 카테고리는 필수입니다.");
    Objects.requireNonNull(transactionAt, "거래일시는 필수입니다.");
    Objects.requireNonNull(transactionType, "거래 유형은 필수입니다.");

    if (merchantName == null || merchantName.isBlank()) {
        throw new IllegalArgumentException("사용처는 필수입니다.");
    }

    if (amount <= 0) {
        throw new IllegalArgumentException("금액은 0보다 커야 합니다.");
    }
}
```

생성자에서는 `user`도 필수 값이므로 별도로 검증했습니다.

```java
Objects.requireNonNull(user, "사용자는 필수입니다.");
```

하지만 `updateTransaction()`에서는 `user`를 검증 대상에 포함하지 않았습니다.

이유는 `updateTransaction()`이 거래 내역의 소유자인 사용자를 변경하는 메서드가 아니기 때문입니다.

거래 내역의 사용자는 생성 시점에 결정되는 값입니다.

거래 내역 수정 시점에는 카테고리, 거래일시, 사용처, 금액, 거래 유형, 지출 합계 포함 여부, 메모 정도만 변경됩니다.

따라서 생성 시점 검증과 수정 시점 검증을 구분했습니다.

```text
생성 시점
→ user 포함 검증
→ 거래 내역이 누구의 것인지 반드시 필요

수정 시점
→ user 제외
→ 거래 소유자를 변경하는 메서드가 아니기 때문
```

이 기준을 세우고 나니 검증 코드도 조금 더 자연스러워졌습니다.

무조건 모든 필드를 매번 검증하는 것이 아니라, 해당 메서드가 실제로 변경하는 값과 그 시점에 필요한 도메인 규칙을 기준으로 검증해야 한다고 봤습니다.

참고로 `Objects.requireNonNull()`은 null일 때 `NullPointerException`을 발생시킵니다.

예외 타입까지 `IllegalArgumentException`으로 통일하고 싶다면 다음처럼 직접 작성할 수도 있습니다.

```java
if (transactionCategory == null) {
    throw new IllegalArgumentException("거래 카테고리는 필수입니다.");
}
```

이번 단계에서는 핵심이 커스텀 예외 타입을 정교하게 나누는 것보다는, Entity가 잘못된 상태로 변경되지 않게 막는 것이라고 봤습니다.

그래서 null 검증에는 `Objects.requireNonNull()`을 사용하고, 문자열 공백이나 금액 범위 검증에는 `IllegalArgumentException`을 사용했습니다.

## User에 적용한 검증

`User`에서는 `profileImageUrl`도 필수값으로 보기로 했습니다.

기존에는 선택 값처럼 볼 수도 있었지만, 현재 프로젝트에서는 기본 프로필 이미지 URL을 포함해서 항상 프로필 이미지 URL이 존재하는 구조로 판단했습니다.

그래서 컬럼도 `nullable = false`로 변경했습니다.

```java
@Column(name = "profile_image_url", nullable = false, length = 500)
private String profileImageUrl;
```

프로필 수정 메서드에서는 닉네임과 프로필 이미지 URL을 함께 검증했습니다.

```java
public void updateProfile(String nickname, String profileImageUrl) {
    validateProfile(nickname, profileImageUrl);

    this.nickname = nickname;
    this.profileImageUrl = profileImageUrl;
}
```

검증 메서드는 Entity 내부 private 메서드로 두었습니다.

```java
private void validateProfile(String nickname, String profileImageUrl) {
    if (nickname == null || nickname.isBlank()) {
        throw new IllegalArgumentException("닉네임은 필수입니다.");
    }

    if (profileImageUrl == null || profileImageUrl.isBlank()) {
        throw new IllegalArgumentException("프로필 이미지 URL은 필수입니다.");
    }
}
```

여기서도 DTO Validation과 역할을 나누어 생각했습니다.

사용자가 닉네임을 비워서 요청했다면 DTO에서 친절한 검증 메시지를 내려줄 수 있습니다.

하지만 Service 내부에서 잘못된 값으로 `updateProfile()`을 호출한다면, Entity가 스스로 잘못된 상태를 막아야 합니다.

이렇게 보면 `User` 내부의 검증은 중복이라기보다 `User`라는 도메인 객체가 가져야 할 최소한의 자기 보호에 가깝다고 판단했습니다.

## Entity 내부 예외는 커스텀 예외로 처리해야 할까?

![예외 처리 책임 분리](https://velog.velcdn.com/images/mindoli/post/35efa50e-5731-421d-b915-412f7fbc2ce2/image.png)


Entity 내부 검증을 추가하면서 또 하나 고민한 부분이 있었습니다.

> Entity 내부에서 `IllegalArgumentException`을 던질지, 프로젝트의 커스텀 예외 구조를 사용할지

페이메모리 프로젝트에는 공통 예외 구조가 있었습니다.

```java
public interface BaseException {
    String getExceptionName();
    HttpStatus getHttpStatus();
    String getMessage();
}
```

```java
@RequiredArgsConstructor
public class GeneralException extends RuntimeException {

    private final BaseException exception;

    public String getExceptionName() {
        return exception.getExceptionName();
    }

    public HttpStatus getHttpStatus() {
        return exception.getHttpStatus();
    }

    public String getMessage() {
        return exception.getMessage();
    }
}
```

처음에는 Entity 내부 검증도 이 구조를 따라야 하는지 고민했습니다.

예를 들어 금액이 0 이하일 때 다음처럼 도메인 Exception Enum을 만들 수도 있습니다.

```java
throw new GeneralException(PaymentTransactionException.INVALID_AMOUNT);
```

이렇게 하면 예외 응답 형식을 통일할 수 있다는 장점이 있습니다.

하지만 이번 단계에서는 Entity 내부 검증의 목적을 다시 생각했습니다.

Entity 내부 검증은 사용자에게 직접 응답을 내려주기 위한 검증이라기보다, 잘못된 도메인 상태를 방지하기 위한 방어 코드에 가까웠습니다.

사용자가 잘못된 요청을 보냈을 때 친절한 메시지를 내려주는 역할은 Request DTO Validation에서 처리하는 것이 더 자연스럽다고 봤습니다.

반면 Entity 내부에서 발생하는 예외는 개발 중 잘못된 사용을 빠르게 발견하기 위한 성격이 더 강하다고 판단했습니다.

그래서 이번 프로젝트에서는 다음처럼 정리했습니다.

```text
DTO Validation
→ 사용자 입력 검증
→ 한글 메시지
→ 클라이언트에게 전달할 수 있는 검증 오류

Entity 내부 검증
→ 도메인 상태 보호
→ IllegalArgumentException 또는 기본 런타임 예외
→ 개발 중 잘못된 사용을 빠르게 발견하기 위한 방어 코드

Service 비즈니스 예외
→ GeneralException + 도메인 Exception Enum
→ 존재하지 않는 사용자, 권한 없음, 중복 요청 등 비즈니스 규칙 위반 처리
```

예를 들어 사용자가 존재하지 않는 거래 내역을 수정하려고 한다면, 이는 Entity 내부 검증보다는 Service 계층의 비즈니스 예외에 가깝습니다.

```java
throw new GeneralException(PaymentTransactionException.NOT_FOUND);
```

반면 `PaymentTransaction`의 `amount`가 0 이하로 들어오는 것은 Entity가 유효하지 않은 상태가 되는 문제에 가깝습니다.

```java
if (amount <= 0) {
    throw new IllegalArgumentException("금액은 0보다 커야 합니다.");
}
```

이번 단계에서는 Entity 내부까지 모두 커스텀 예외로 감싸기보다, 계층별 예외의 목적을 나누는 편이 더 단순하고 명확하다고 봤습니다.

물론 이 방식이 무조건 정답이라는 뜻은 아닙니다.

Entity 내부 예외까지 모두 도메인 예외로 통일하는 팀도 있을 수 있습니다.

다만 현재 페이메모리 프로젝트의 규모와 단계에서는 Entity 내부 검증은 기본 런타임 예외로 두고, 클라이언트 응답과 직접 관련된 예외는 DTO Validation과 Service 비즈니스 예외에서 처리하는 기준이 더 적절하다고 판단했습니다.

## 검증 메서드는 Entity 내부에 둘까, 별도 클래스로 분리할까?

검증 메서드를 추가하다 보니 또 다른 고민이 생겼습니다.

> `validateTransactionFields()` 같은 메서드를 별도 Validator 클래스로 빼야 할까?

예를 들어 다음과 같은 클래스를 만들 수도 있습니다.

```java
public class PaymentTransactionValidator {

    public static void validateTransactionFields(
            TransactionCategory transactionCategory,
            Instant transactionAt,
            String merchantName,
            long amount,
            TransactionType transactionType
    ) {
        // validation
    }
}
```

하지만 현재 검증은 `PaymentTransaction` 내부에서만 사용하는 필수 필드 검증이었습니다.

검증 규칙도 아직 복잡하지 않았습니다.

- 거래 카테고리는 필수입니다.
- 거래 일시는 필수입니다.
- 거래 유형은 필수입니다.
- 사용처는 비어 있을 수 없습니다.
- 금액은 0보다 커야 합니다.

이 정도의 규칙은 `PaymentTransaction` 안에 private 메서드로 두는 편이 응집도가 더 높다고 봤습니다.

```java
private void validateTransactionFields(
        TransactionCategory transactionCategory,
        Instant transactionAt,
        String merchantName,
        long amount,
        TransactionType transactionType
) {
    Objects.requireNonNull(transactionCategory, "거래 카테고리는 필수입니다.");
    Objects.requireNonNull(transactionAt, "거래일시는 필수입니다.");
    Objects.requireNonNull(transactionType, "거래 유형은 필수입니다.");

    if (merchantName == null || merchantName.isBlank()) {
        throw new IllegalArgumentException("사용처는 필수입니다.");
    }

    if (amount <= 0) {
        throw new IllegalArgumentException("금액은 0보다 커야 합니다.");
    }
}
```

검증 로직이 Entity의 상태와 밀접하게 연결되어 있다면, Entity 내부에 두는 것이 읽기도 쉬웠습니다.

`updateTransaction()`을 읽을 때 바로 위나 아래에서 어떤 값들이 필수인지 확인할 수 있기 때문입니다.

이번 단계에서는 별도 Validator 클래스로 분리하지 않았습니다.

다만 앞으로 다음과 같은 상황이 생긴다면 분리를 다시 고민할 수 있다고 정리했습니다.

```text
여러 Entity에서 같은 검증 로직을 반복해서 사용합니다.

검증 규칙이 복잡해집니다.

별도 테스트가 필요할 정도로 정책이 커집니다.

금액, 기간, 카테고리 같은 값이 Value Object로 분리됩니다.
```

예를 들어 금액 검증이 단순히 `0보다 커야 한다`에서 끝나지 않고, 통화, 최소 금액, 최대 금액, 반올림 정책, 정산 정책까지 포함하게 된다면 `Money` 같은 Value Object로 분리하는 것이 더 나을 수 있습니다.

```java
public class Money {

    private final long amount;

    public Money(long amount) {
        if (amount <= 0) {
            throw new IllegalArgumentException("금액은 0보다 커야 합니다.");
        }

        this.amount = amount;
    }
}
```

이렇게 되면 검증 책임이 Entity에서 Value Object로 자연스럽게 이동할 수 있습니다.

하지만 현재 단계에서는 Entity 내부 private 메서드가 가장 단순하고 적절한 선택이라고 봤습니다.

## 최종적으로 적용한 기준

이번 리뷰를 반영하면서 최종적으로 적용한 기준은 다음과 같았습니다.

### 1. Request DTO에서는 Wrapper 타입과 Bean Validation을 사용했습니다

외부 요청에서는 값이 누락될 수 있습니다.

따라서 누락 여부를 검증하려면 primitive 타입보다 Wrapper 타입이 더 적절했습니다.

```java
public record CreatePaymentTransactionRequest(
        @NotNull(message = "금액은 필수입니다.")
        @Positive(message = "금액은 0보다 커야 합니다.")
        Long amount
) {
}
```

`Long`을 사용하면 사용자가 아예 값을 보내지 않았을 때 null로 받을 수 있고, `@NotNull`을 통해 명확한 검증 메시지를 줄 수 있습니다.

`boolean`도 마찬가지입니다.

요청에서 사용자가 값을 보내지 않았는지 검증해야 한다면 DTO에서는 `Boolean`을 사용할 수 있습니다.

```java
public record UpdatePaymentTransactionRequest(
        @NotNull(message = "지출 합계 포함 여부는 필수입니다.")
        Boolean includedInSpending
) {
}
```

### 2. Entity에서는 null 상태가 없어야 하는 필드에 primitive 타입을 사용했습니다

Entity 내부에서 null이 의미 없는 값이라면 primitive 타입을 사용했습니다.

```java
@Column(name = "amount", nullable = false)
private long amount;
```

```java
@Column(name = "included_in_spending", nullable = false)
private boolean includedInSpending;
```

이렇게 하면 Entity에서 “값이 없음”이라는 상태를 표현하지 않습니다.

대신 primitive 타입의 기본값이 도메인 규칙에 어긋날 수 있는 경우에는 별도 검증을 두었습니다.

```java
if (amount <= 0) {
    throw new IllegalArgumentException("금액은 0보다 커야 합니다.");
}
```

### 3. Entity 내부에서는 필수 값에 대한 최소한의 방어 검증을 두었습니다

`nullable = false` 필드는 DB 저장 시점뿐만 아니라 Entity 상태 자체에서도 보호할 필요가 있다고 봤습니다.

```java
private void validateTransactionFields(
        TransactionCategory transactionCategory,
        Instant transactionAt,
        String merchantName,
        long amount,
        TransactionType transactionType
) {
    Objects.requireNonNull(transactionCategory, "거래 카테고리는 필수입니다.");
    Objects.requireNonNull(transactionAt, "거래일시는 필수입니다.");
    Objects.requireNonNull(transactionType, "거래 유형은 필수입니다.");

    if (merchantName == null || merchantName.isBlank()) {
        throw new IllegalArgumentException("사용처는 필수입니다.");
    }

    if (amount <= 0) {
        throw new IllegalArgumentException("금액은 0보다 커야 합니다.");
    }
}
```

### 4. 생성 시점 검증과 수정 시점 검증을 구분했습니다

생성할 때 반드시 필요한 값과 수정할 때 변경 가능한 값은 다를 수 있습니다.

`PaymentTransaction`에서는 생성 시점에 `user`가 필수였습니다.

하지만 `updateTransaction()`은 거래 소유자를 변경하는 메서드가 아니므로 `user`는 검증 대상에서 제외했습니다.

```text
생성자
→ user 검증 포함

updateTransaction()
→ user 검증 제외
→ 수정 대상 필드만 검증
```

### 5. Entity 내부 검증은 현재 단계에서는 기본 런타임 예외로 처리했습니다

사용자 입력 오류는 DTO Validation에서 처리했습니다.

비즈니스 규칙 위반은 Service 계층에서 `GeneralException`과 도메인 Exception Enum으로 처리했습니다.

Entity 내부 검증은 잘못된 도메인 상태를 빠르게 발견하기 위한 방어 코드로 보고, 현재 단계에서는 `IllegalArgumentException` 또는 `Objects.requireNonNull()`을 사용했습니다.

```java
if (merchantName == null || merchantName.isBlank()) {
    throw new IllegalArgumentException("사용처는 필수입니다.");
}
```

### 6. 단순한 검증은 Entity 내부 private 메서드로 유지했습니다

현재 검증 로직은 특정 Entity 내부에서만 사용되며 복잡하지 않았습니다.

따라서 별도 Validator 클래스로 분리하지 않고, Entity 내부 private 메서드로 유지했습니다.

검증 로직이 커지거나 여러 Entity에서 반복되면 그때 분리하는 방향이 더 자연스럽다고 봤습니다.

## 이번 리뷰를 통해 배운 점

이번 리뷰를 통해 가장 크게 배운 점은 `nullable = false`를 단순히 DB 컬럼 옵션으로만 보면 안 된다는 점이었습니다.

처음에는 `nullable = false`를 붙였으니 DB에서 막아줄 것이라고 생각했습니다.

그리고 요청 값은 DTO에서 검증하니 Entity 내부 검증은 중복이라고 생각했습니다.

하지만 Entity는 단순히 DB 테이블과 매핑되는 객체만은 아니었습니다.

도메인 상태를 표현하는 객체였습니다.

그렇다면 Entity는 자기 자신이 유효하지 않은 상태가 되지 않도록 최소한의 책임을 가져야 한다고 봤습니다.

또 하나 배운 점은 DTO와 Entity의 타입 선택 기준이었습니다.

처음에는 습관적으로 `Long`, `Boolean` 같은 Wrapper 타입을 사용했습니다.

하지만 요청 DTO와 Entity는 목적이 달랐습니다.

DTO에서는 값이 누락되었는지 확인해야 하므로 Wrapper 타입이 유용했습니다.

반면 Entity에서는 null 상태가 존재하면 안 되는 값이라면 primitive 타입이 더 명확할 수 있었습니다.

```text
Request DTO
→ Long, Boolean
→ null을 통해 누락 여부를 검증

Entity
→ long, boolean
→ null 상태를 허용하지 않음
```

예외 처리도 마찬가지였습니다.

프로젝트에 공통 예외 구조가 있다고 해서 모든 예외를 무조건 커스텀 예외로 감싸야 하는 것은 아니라고 봤습니다.

예외도 계층별 목적에 따라 다르게 바라볼 필요가 있었습니다.

```text
DTO Validation
→ 사용자가 잘못 입력한 값

Entity 내부 검증
→ 잘못된 도메인 상태

Service 비즈니스 예외
→ 비즈니스 규칙 위반
```

이번 프로젝트에서는 Entity 내부 검증을 사용자 응답을 위한 예외라기보다 개발 중 잘못된 사용을 빠르게 발견하기 위한 방어 코드로 정리했습니다.

그래서 현재 단계에서는 커스텀 예외보다 기본 런타임 예외를 사용하는 쪽이 더 단순하고 적절하다고 판단했습니다.

무엇보다 이번 리뷰는 “검증 코드를 어디에 둘 것인가”를 단순히 코드 중복 여부로만 판단하면 안 된다는 점을 알려주었습니다.

같은 값 검증처럼 보여도, 어떤 계층에서 어떤 목적을 가지고 검증하는지에 따라 의미가 달라졌습니다.

## 정리하면

![최종 기준 요약](https://velog.velcdn.com/images/mindoli/post/5a96376b-317c-49c8-93d7-23298ef90dc0/image.png)

이번 고민은 한 문장으로 정리할 수 있었습니다.

> DTO Validation은 사용자 입력을 검증하고, Entity 내부 검증은 도메인 객체의 유효한 상태를 보호했습니다.

처음에는 DTO에서 이미 검증한 값을 Entity에서 다시 검증하는 것이 중복이라고 생각했습니다.

하지만 Entity는 Controller 요청을 통해서만 생성되거나 수정되지 않았습니다.

Service 내부 로직, 테스트 코드, 배치 작업, 관리자 기능, 초기 데이터 생성 코드, 외부 API 연동, AI 분석 결과 기반 자동 생성 등 다양한 경로에서 Entity가 사용될 수 있었습니다.

따라서 이번 프로젝트에서는 `nullable = false` 필드에 대해 Entity 내부에서도 최소한의 방어 검증을 두기로 했습니다.

또한 타입 선택 기준도 함께 정리했습니다.

```text
외부 요청에서 값이 누락되었는지 검증해야 하는 DTO
→ Wrapper 타입 사용
→ Long, Boolean
→ @NotNull, @Positive 등 Bean Validation 사용

도메인 내부에서 null 상태가 존재하면 안 되는 Entity
→ primitive 타입 사용
→ long, boolean
→ 필요한 경우 Entity 내부에서 범위 검증
```

예외 처리 기준은 다음과 같이 나누었습니다.

```text
DTO Validation
→ 사용자 입력 검증
→ 한글 메시지
→ 클라이언트에게 전달할 수 있는 검증 오류

Entity 내부 검증
→ 도메인 상태 보호
→ IllegalArgumentException 또는 기본 런타임 예외
→ 개발 중 잘못된 사용을 빠르게 발견하기 위한 방어 코드

Service 비즈니스 예외
→ GeneralException + 도메인 Exception Enum
→ 존재하지 않는 사용자, 권한 없음, 중복 요청 등 비즈니스 규칙 위반 처리
```

검증 메서드는 현재 단계에서는 Entity 내부 private 메서드로 유지했습니다.

아직 검증 규칙이 복잡하지 않고, 특정 Entity에 강하게 묶여 있는 규칙이기 때문입니다.

다만 검증 로직이 여러 Entity에서 반복되거나, 정책이 복잡해지거나, 값 자체가 독립적인 의미를 가지게 된다면 Validator 또는 Value Object로 분리할 수 있다고 봤습니다.

이번 방식이 무조건 정답이라고 생각하지는 않았습니다.

팀의 예외 처리 방식, 도메인 복잡도, Entity 설계 방식에 따라 다른 선택을 할 수 있습니다.

다만 현재 페이메모리 프로젝트 단계에서는 다음 기준이 가장 적절하다고 판단했습니다.

```text
요청 값은 DTO에서 친절하게 검증했습니다.
Entity는 자기 자신의 필수 상태를 방어했습니다.
DB의 nullable = false는 마지막 방어선으로 두었습니다.
비즈니스 규칙 위반은 Service 계층의 커스텀 예외로 처리했습니다.
```

이번 PR 리뷰를 통해 단순히 코드를 수정한 것이 아니라, Entity를 어떻게 바라볼지에 대한 기준을 조금 더 명확히 세울 수 있었습니다.
